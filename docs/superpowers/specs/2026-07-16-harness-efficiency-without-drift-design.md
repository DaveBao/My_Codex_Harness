# Harness 效率优化与功能防漂移设计

## 问题

Harness 通过有界上下文、相互隔离的 Builder 工作树、独立的 Reviewer 验收、串行集成以及合并后的全局验证来保证功能忠实度。当前基于 Codex App 原生能力的实现是正确的，但在 Orchestrator 的流程衔接和重复的大范围读取上消耗了本可避免的时间与上下文。

已完成的个人财务管理系统 F001/F002 运行提供了以下证据：

- F002 经历了三轮 Builder/Reviewer。重试是必要的：Reviewer 发现了两个 Builder 测试未覆盖的真实验收失败。
- F002 在已测量阶段之间耗时 19 分 10.404 秒；该时间不包括 Builder、Reviewer、合并、验证、清理及 Librarian 工作。
- F002 的六次 Builder/Reviewer 调用在开始探索源代码前，至少重复载入了 106,483 字节的角色合同、功能上下文、项目地图、Codex 策略及选定的交接数据。
- 当前每个活跃功能都引用同一个通用锚点 `docs/project-map.md#entry-points`，因此 Builder 和 Reviewer 必须读取完整项目地图才能找到领域导航信息。
- Orchestrator 合同使用文字描述确定性的闭包检查、生命周期事件构造、计时及追加规则。当前主 App 任务需要手动执行这些机械步骤。
- Codex App 不向 Orchestrator 提供官方的逐角色 token 用量。因此，必须通过确定性的输入字节数及调用次数代理指标验证 token 降幅，同时继续将生命周期事件中的 token 值保留为 `null`。

## 目标

1. 在等同于 F002、包含三轮 Builder/Reviewer 的基准测试中，将 Harness 控制的上下文字节数至少降低 30%。
2. 在基于 Codex App 原生能力的冒烟运行中，将非角色编排间隔至少降低 25%。
3. 从日常 Orchestrator 操作中移除手工生命周期事件构造和耗时计算。
4. 保持 Builder、Reviewer、Librarian 和 Orchestrator 的职责不变。
5. 保持功能忠实度：不得削弱任何验收标准、验证门、重试隔离、证据要求或集成检查。
6. 继续使用 Codex App 原生子任务；不得通过 Codex CLI 调用 Builder 或 Reviewer。
7. 不新增第三方运行时依赖。

## 非目标

- 根据文本长度、价格或代理字节数估算无法取得的 token 用量。
- 正式重试时复用已经失败的 Builder 子任务。
- 合并 Builder 与 Reviewer，或允许 Reviewer 在没有独立证据的情况下信任 Builder 的结果。
- 移除 TDD、交接验证、全局验证、工作树隔离或已提交上下文检查。
- 改变 `TODO.json` 的状态所有权，或让遥测数据成为权威工作流状态。
- 在本阶段重新设计 Builder 交接 schema。
- 自动提交、推送、归档、删除脏工作树，或跨越 Owner 审批门。

## 约束与不变量

以下规则继续作为硬性不变量：

- `TODO.json` 仍是唯一的活跃工作流状态来源；只有 Orchestrator 可以修改状态和尝试次数。
- 正式重试必须启动新的 Builder App 子任务，并且只能复用已经记录的分支、工作树、代码以及确切的持久化返工交接。
- 只有上下文修复可以恢复同一个 Builder 任务。
- Reviewer 必须用可观察证据独立验证每一项验收标准。
- Reviewer 不得修改产品代码或实现测试。
- 只有 Reviewer 通过、被审查的提交完成集成且项目级验证通过后，功能才能标记为通过。
- 在派发任务前，静态闭包以及所有已披露的功能引用都必须存在于指定的基础提交中，并与当前文件字节一致。
- 上下文切片必须保持原文并通过哈希验证。不得用模型生成的摘要替代权威源文本。
- 缺失、歧义、陈旧或不安全的上下文必须安全失败，或回退到现有的完整文档读取；绝不能静默遗漏上下文。
- 每个生命周期 span 保留一个 `started` 事件和且仅有一个终态 `finished` 事件，并记录实测 `durationMs`。
- App 原生角色的 token 字段保持为 `null`，原因使用 `app_usage_unavailable`。

## 选定方案

使用确定性控制辅助程序处理 Orchestrator 的本地机械操作，并扩展现有上下文辅助程序，以支持保持原文且带哈希地址的上下文切片。更新规划规则，使未来的功能包含精确引用。所有语义判断仍由现有角色负责。

相比只修改提示词，该方案能够让节省效果可重复、可测试。相比合并或减少验证，该方案更可取，因为 F002 已证明独立 Reviewer 正在阻止真实的功能漂移。

## 架构

### 1. 确定性 Orchestrator 控制辅助程序

新增不依赖第三方库的 Python 命令行辅助程序 `skills/orchestrator/scripts/harness_control.py`。它只负责以下确定性的本地机械操作：

- 针对指定基础 SHA，预检静态/运行时闭包及功能引用是否逐字节一致；
- 创建并追加生命周期 `started` 事件；
- 创建生命周期终态事件，根据持久化的匹配起始事件计算耗时，并追加事件；
- 对其创建的事件执行生命周期身份、配对及有界字段验证；
- 输出精简的上下文字节基准报告。

它不得：

- 选择功能或 wave；
- 解释验收结果；
- 更新 `TODO.json`；
- 判断重试、合并、通过、阻塞或暂停在语义上是否正确；
- 创建、移除、合并或重置工作树；
- 启动角色；
- 提交或推送。

该辅助程序只在 Harness 已被明确激活的运行期间写入 `worklog/logs/lifecycle.jsonl`。它使用带追加语义的 `os.open`，验证目标是主控制根目录内的普通非符号链接文件，写入一个精简 JSON 对象和换行符，并在返回前刷新数据。终态命令必须拒绝缺失、重复、已经终止或不匹配的 span。

该辅助程序只输出包含标识符、时间戳、实测耗时和状态的有界响应，不会将完整生命周期日志或无关状态回显到 Orchestrator 上下文中。

### 2. 精确上下文分配

扩展 `skills/orchestrator/scripts/harness_context.py`，但不改变 `harness-featurespec-v1` 的哈希语义。

新增 assignment 视图，返回：

- 用于计算 `featureSpecSha256` 的不可变功能定义字段；
- 规范功能哈希及版本；
- 经过规范化的精确引用路径及锚点；
- 不包含任何同级功能数据；
- 除非调用方明确请求某个确切的现有交接，否则不包含可变状态、尝试次数、交接历史列表或验证历史。

针对显式的 `path#anchor` 引用新增 Markdown 章节解析能力：

- 切片前，将完整文件字节与指定基础 SHA 进行比较；
- 使用稳定的 GitHub 风格锚点规范化规则，只解析唯一一个标题；
- 原样返回该标题及其正文，直到下一个同级或更高级标题为止；
- 包含完整文件的 SHA-256 和返回的字节数；
- 拒绝逃逸路径、指向根目录外的符号链接、缺失文件、重复匹配标题和格式错误的锚点；
- 当旧功能只有无法安全提供必要导航的通用引用时，回退到现有的完整文档读取。

Builder 和 Reviewer 继续只解析 Orchestrator 指定的确切交接事件 ID。正式重试接收最新且相关的失败 verdict，以及理解该 verdict 所需的上一份交接；不得读取完整功能历史。

### 3. 精确的规划引用

更新 `skills/to-exec-plan/SKILL.md`，使新生成的功能引用现有权威内容中范围最小的章节：

- 如果已经存在具体的项目地图领域标题，则引用该标题；
- 引用具体的 PRD 决策及测试章节；
- 只有当已确认代码或合同路径确实是稳定入口点时，才引用这些路径；
- 不得将 `docs/project-map.md#entry-points` 作为通用占位符；
- 如果不存在精确的项目地图章节，则保留明确的产品/合同引用并报告导航缺口，而不是虚构标题。

这些规则只影响新功能或被明确重新规划且仍处于 pending 状态的功能。现有的活跃、执行中、已通过、失败或阻塞的功能定义不会被静默重写，因为修改不可变输入会使其分配哈希失效。

### 4. 更精简的核心合同

在每个角色的 `SKILL.md` 中保留激活、所有权、结果、重试及忠实度不变量。用精确的辅助程序命令和预期结果码，替换 Orchestrator 中重复的机械生命周期/预检说明。低频恢复细节可以移入直接链接的参考文件，仅在发生对应失败时由 Orchestrator 读取。

不得仅为了减少字节数而删除规则。只有满足以下任一条件时，规则才可以移出核心 Skill：

- 确定性辅助程序能够强制执行该规则，并且测试已证明这种强制执行；或
- 核心 Skill 明确写出一个精确的条件式引用及其触发条件。

Builder、Reviewer 和 Librarian 的合同继续相互独立，不进行合并。

### 5. 立即安全推进

收到角色结果后，Orchestrator 必须在同一个活跃回合中结束角色 span、验证并持久化结果，然后启动下一个无需审批的安全操作。在 Builder 交接、Reviewer 派发、合并、全局验证、清理、Librarian、checkpoint 以及下一个依赖就绪 wave 之间，不得等待 Owner 再发一条消息。

现有审批门、终态阻塞、安全风险操作、规格缺口、重试耗尽以及项目完成归档仍必须由 Owner 参与。

## 数据流

1. Owner 明确启动或恢复 Harness。
2. Orchestrator 从权威 TODO 状态中选择依赖就绪的功能。
3. `harness_control.py preflight` 验证已提交闭包及已披露的确切引用。
4. `harness_context.py assignment` 返回不可变功能视图及规范哈希。
5. 从权威文件中解析确切的 Markdown 章节及交接事件。旧版歧义引用使用完整文档回退。
6. Orchestrator 使用引用而非嵌入式源文本派发 App 原生 Builder。
7. Builder 独立打开有界上下文，完成实现、验证和本地提交，并返回保持不变的规范化交接合同。
8. Orchestrator 对交接执行机械预检并持久化，然后立即派发 Reviewer。
9. Reviewer 独立执行每一项验收标准，并返回证据和 verdict。
10. 审查失败时创建可见的持久化返工事件并启动新的 Builder 子任务；审查通过后进入串行集成和全局验证。
11. Librarian 和 checkpoint 处理继续保持 wave 级作用域。
12. `harness_control.py` 使用经过验证的生命周期事件包围每个受跟踪的本地或角色操作。

## 错误处理

- 不安全路径、符号链接逃逸、基础提交不匹配、schema 不匹配、重复 span 或格式错误的 JSON 必须返回有界的符号错误，并且不执行写入。
- 生命周期追加失败时不得留下部分 JSON 对象。如果无法安全恢复遥测完整性，Orchestrator 必须写入 checkpoint 并暂停。
- 缺失确切 Markdown 锚点时返回符号化上下文错误。旧版通用引用可以使用完整文档回退；新的精确引用必须安全失败，使规划漂移变得可见。
- 辅助程序版本不匹配或输出异常属于 Harness 完整性失败，不消耗功能尝试次数。
- 无效的 Builder 或 Reviewer 结果继续使用现有重试规则；控制辅助程序不得重新解释这些结果。
- 合并后的全局验证失败，在返工前仍然必须证明能够安全恢复。

## 验证策略

### 合同测试

新增测试以证明：

- `harness_control.py` 能创建符合 schema 的精简 started/finished 事件对，并根据持久化时间戳计算耗时；
- 重复、缺失、不匹配、已终止、符号链接、非主根目录及格式错误的生命周期输入会在不修改数据的情况下失败；
- preflight 能检测缺失或字节发生漂移的闭包/引用路径；
- assignment 输出包含不可变功能输入，但排除同级功能和可变历史；
- Markdown 切片保持原文、带有哈希地址、结果稳定，并在正确的标题边界停止；
- 缺失/重复锚点及不安全路径会安全失败；
- 旧版通用引用使用已记录的完整文件回退；
- `to-exec-plan` 要求精确引用，并拒绝将通用占位符作为指导；
- 源码、打包产物、脚手架、安装副本和用户级副本保持同步。

### 功能忠实度测试

保留并加强测试以证明：

- Reviewer 仍是独立的 App 原生任务；
- 正式重试仍启动新的 Builder 任务；
- 只有上下文修复继续使用同一个 Builder 任务；
- assignment 切片前后的验收标准及不可变功能哈希完全相同；
- Builder/Reviewer 无法读取同级功能，也无法自行推断最新交接；
- 功能通过仍然要求 Reviewer verdict、确切的提交身份、合并及全局验证；
- 遥测数据不能改变 TODO 状态；
- 无法取得的 App 用量继续保持为 `null`。

### 效率基准

创建一个确定性的 F002 形态 fixture，其中包含三轮 Builder/Reviewer。只计算由 Harness 控制的字节数：

- 角色 Skill 字节数；
- 不可变 assignment 字节数；
- 已解析的项目地图及策略字节数；
- 精确指定的交接/verdict 字节数；
- 有界的控制辅助程序输出字节数。

基线：在已完成的 F002 运行中，六次 Builder/Reviewer 调用共计 106,483 字节；不包括系统提示词、源代码、测试输出以及无法取得的官方 token 数量。

验收阈值：优化后的 fixture 不超过 74,538 字节，即至少降低 30%，并保持相同的六次角色调用及相同的三次审查结果。

运行一次 App 原生冒烟工作流并比较生命周期遥测。非角色编排间隔相较同等基线必须至少改善 25%。如果 App 调度噪声使比较无法得出可靠结论，应将时间结果报告为“无法确定”，而不是宣称成功；字节及合同基准仍然是确定性的。

### 仓库验证

运行：

```bash
python3 -m unittest discover -s tests -v
python3 scripts/validate_package.py
python3 scripts/doctor.py --source .
```

此外，还应在临时输出目录中构建 bundle，安装到临时 home，初始化一个临时项目，并验证源码、打包产物、脚手架及已安装辅助程序/合同的哈希。

## 推出与兼容性

1. 在源码仓库中实现并测试，不修改正在运行的真实项目。
2. 保持现有 `harness-featurespec-v1` 哈希以及 Builder 交接/生命周期 schema 版本 1。
3. 为现有 TODO 引用保留旧版完整文档回退。
4. 只有在新增辅助程序或合同行为确有需要时，才更新包验证、bundle 内容、安装程序、doctor、README、CHANGELOG 和脚手架。
5. 只有仓库测试通过且 Owner 授权安装步骤后，才将经过验证的包安装到用户级位置。
6. 在个人财务管理系统使用优化版 Harness 前，先通过一次性初始化仓库进行验证。

回滚方式是降级包版本，或通过现有安装程序 journal 恢复。不需要迁移项目 TODO。

## 成功标准

- 确定性基准中的上下文字节数至少降低 30%。
- 同等 App 原生冒烟运行中的非角色编排间隔至少降低 25%；如果受运行时噪声影响，则必须明确报告为无法确定。
- 所有现有及新增的包、合同、安装程序、脚手架及辅助程序测试全部通过。
- 不削弱任何功能哈希、验收标准、重试规则、角色所有权边界、证据要求或全局验证门。
- 不伪造官方 token 数量；App 角色 token 字段继续保持为 `null`。
- 在宣布优化完成前，源码、打包产物、脚手架、安装副本及用户级产物必须完成同步并通过哈希验证。

## 开放问题

无。Owner 已于 2026-07-16 批准“确定性控制 + 精确上下文切片”方案。由于无法取得 App 原生角色的官方用量数据，运行时 token 降幅将通过已批准的确定性代理指标进行评估。
